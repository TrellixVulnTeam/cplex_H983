CPXXsetlazyconstraintcallbackfunc and CPXsetlazyconstraintcallbackfunc

The routine CPXXsetlazyconstraintcallbackfunc/CPXsetlazyconstraintcallbackfunc sets and modifies the user-written callback for adding lazy constraints.

int CPXXsetlazyconstraintcallbackfunc( CPXENVptr env, int(CPXPUBLIC *lazyconcallback)(CPXX_CALLBACK_CUT_ARGS), void * cbhandle )

int CPXsetlazyconstraintcallbackfunc( CPXENVptr env, int(CPXPUBLIC *lazyconcallback)(CALLBACK_CUT_ARGS), void * cbhandle )

Description

The routine CPXXsetlazyconstraintcallbackfunc/CPXsetlazyconstraintcallbackfunc sets and modifies the user-written callback for adding lazy constraints.

Warning:

This is an advanced routine. Advanced routines typically demand a thorough understanding of the algorithms used by CPLEX. Thus they incur a higher risk of incorrect behavior in your application, behavior that can be difficult to debug. Therefore, the team encourages you to consider carefully whether you can accomplish the same task by means of other Callable Library routines instead.

The user-written callback is called by CPLEX in these situations:

  • when CPLEX compares an integer-feasible solution (including an integer-feasible solution provided by a MIP start before any nodes exist) to lazy constraints;
  • when the LP at a node is unbounded, and a lazy constraint might cut off the primal ray.

Within the user-written callback, the routine CPXXgetcallbacknodelp/CPXgetcallbacknodelp and other query routines from the Callable Library access information about the subproblem.

Within the user-written callback, the routine CPXXcutcallbackadd/CPXcutcallbackadd can be used to add lazy constraints globally to the current node LP subproblem and global tree during MIP branch and cut. Likewise, the routine CPXXcutcallbackaddlocal/CPXcutcallbackaddlocal can be used to add lazy constraints locally to the current node LP subproblem and associated subtree during MIP branch and cut. Lazy constraints added to the problem are first put into a pool of lazy constraints, so they are not present in the subproblem LP until after the user-written cut callback is finished.

Any lazy constraints that are duplicates of lazy constraints already in the subproblem are not added to the subproblem. Otherwise, the user generated lazy constraints are treated according to the value of the argument purgeable in effect when the lazy constraints were added. See the routine CPXXcutcallbackadd/CPXcutcallbackadd for important details about the argument purgeable.

If lazy constraints have been added, the subproblem is re-solved and evaluated, and, if the LP solution is still integer feasible and not cut off, the lazy constraint callback is called again.

If the model has names, added lazy constraints have names of the form unumber where number is a sequence number among all lazy constraints generated.

When lazy constraints are present in a model, whether added explicitly originally or added later by a lazy constraint callback function, CPLEX turns off dual reduction in preprocessing. In other words, the parameter that controls primal and dual reduction type (CPXPARAM_Preprocessing_Reduce) is effectively set to the value 1 (one) CPX_PREREDUCE_PRIMALONLY or 0 (zero) CPX_PREREDUCE_NOPRIMALORDUAL. Furthermore, since lazy constraints can be represented only in the original space, CPLEX turns off nonlinear reductions in preprocessing as well. That is, the parameter linear reduction switch (CPXPARAM_Preprocessing_Linear) is set to 0 (zero).

Warning:
The lazy constraint callback may be invoked during MIP start processing. In that case CPXgetcallbacknodeinfo/CPXXgetcallbacknodeinfo return CPX_LAZYCONSTRAINTCALLBACK_MIPSTART for CPX_CALLBACK_INFO_LAZY_SOURCE. When this value is returned some special considerations apply:
  • MIP start processing occurs very early in the solution process. At this point no search tree is setup yet and there are no search tree nodes yet. Consequently, a lot of the callback functions that require a node context will fail in this situation.
  • Lazy constraints separated when processing a MIP start will be discarded after that MIP start has been processed. This means that the callback may have to separate the same constraint again for the next MIP start or for a solution that is found later in the solution process.

Arguments

env

A pointer to the CPLEX environment, as returned by CPXXopenCPLEX/CPXopenCPLEX.

lazyconcallback

The pointer to the current user-written lazy constraint callback. If no callback has been set, the pointer evaluates to NULL.

cbhandle

A pointer to user private data. This pointer is passed to the user-written cut callback.

Return

The routine returns 0 (zero) if successful and nonzero if an error occurs.

Example

 status = CPXsetlazyconstraintcallbackfunc(env, cutcallback, cbhandle);

See also the example admipex5.c in the standard distribution of the product.

Callback description

 int CPXPUBLIC
 callback (CPXCENVptr env,
           void *cbdata,
           int wherefrom,
           void *cbhandle,
           int *useraction_p);

Callback return value

The callback returns 0 (zero) if successful and nonzero if an error occurs.

Callback arguments

env

A pointer to the CPLEX environment, as returned by CPXXopenCPLEX/CPXopenCPLEX.

cbdata

A pointer to pass to functions that obtain callback-specific information, such as the number of integer-infeasible values at the current node, or that take callback-specific action, such as to add a cut in a user cut callback.

wherefrom

An integer value reporting where in the optimization this function was called. Its possible values and the corresponding situations are:

Value Symbolic Constant Where the callback was called from
115 CPX_CALLBACK_MIP_CUT_FEAS CPLEX has found an integer feasible solution, and the user has the option to generate a lazy constraint not satisfied by that solution.
116 CPX_CALLBACK_MIP_CUT_UNBD CPLEX has found that the LP relaxation at the current node is primal unbounded, and the user can generate a lazy constraint to cut off the primal ray.

cbhandle

A pointer to private user data.

useraction_p

A pointer to an integer specifying the action for CPLEX to take at the completion of the user callback. The table summarizes possible actions.

Table 1. Actions to be taken after a user-written cut callback
Value Symbolic Constant Action
0 CPX_CALLBACK_DEFAULT Use cuts as added
1 CPX_CALLBACK_FAIL Exit optimization
2 CPX_CALLBACK_SET Use cuts as added
